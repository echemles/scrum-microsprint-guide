      Step 3
      
Sprint Planning

      
Five minutes. Then you build.


      

        

          
1

          
Pick one goal each

          
Les and Mattis each commit to a single Sprint Goal. If you can't say it in one sentence, it's too big.

        

        

          
2

          
Choose 1–3 tickets

          
Pull tickets from your backlog into the sprint. More than 3 tickets in 60 minutes = overscope. Cut now, ship later.

        

        

          
3

          
Write the Definition of Done

          
What does "done" look like for each ticket? If you can't demo it, it doesn't count.

        

        

          
4

          
Identify blockers up front

          
If you already know you're stuck on something, say it now. Solve or cut before the build phase.

        

      


      

        Where do tickets live? Tickets are managed in HFS2S (your kanban + Claude Code workflow). This app is the operating layer — HFS2S is the work surface.

      Step 5
      
Demo Review

      
Five minutes. Show, don't tell.


      

        
The only review that counts is a live demo.

        
Open the app. Run the flow. Show your cofounder what works. If you can't show it, it didn't ship.

      


      

        

          
1

          
Demo the increment

          
Walk through the actual feature in the browser. Not the code. Not the diff. The thing.

        

        

          
2

          
Confirm Definition of Done

          
Did it hit the DoD you wrote in your contract? If not, it carries over.

        

        

          
3

          
Capture feedback

          
What surprised your cofounder? What needs another sprint? Note it for retro.

        

        

          
4

          
Mark the ticket Done in HFS2S

          
Only after demo passes. Attach proof (screenshot, recording, or link).

Terms in Plain Language

        

          
🎯
Sprint Goal
The one thing this hour is for.

          
📋
Sprint Backlog
The tiny list you commit to.

          
✅
Definition of Done
The checklist that prevents fake-done.

          
📦
Increment
The usable thing produced.

          
🔄
Retro
The improvement loop.

        


        
Two-Cofounder Model

        

          
Role A: Builder
  • Owns the sprint goal
  • Writes code / creates deliverables
  • Demos the increment

          
Role B: Scrum Master
  • Keeps time + scope
  • Asks facilitator questions
  • Runs review and retro

        

        
Switch roles each microsprint.


        
Scrum Master Script

        

          
Plan
  • What is the one sprint goal?
  • What 1-3 tasks will get us there?
  • What does done look like?

          
Build
  • Still on track?
  • Scope growing? Cut something.
  • Can you demo this yet?

          
Review
  • Show me the working thing.
  • Does it match the goal?
  • What feedback changes next?

          
Retro
  • What went well?
  • What slowed you down?
  • One change for next time?

        


        
Common Failure Modes

        

          
Too much scope — If you can't demo it in the last 10 min, you scoped too big.

          
Skipping review — No review = no verified output.

          
Skipping retro — You'll repeat the same mistakes.

          
Activity ≠ output — Lines of code ≠ shipped increment.

          
One person dominates — Both founders must own a deliverable.
